 PRACTICA 1 – Sincronización con Rsync
# Primero instalo rsync en ambos servidores
sudo dnf install rsync -y

# En el servidor principal (192.168.0.199) creo la carpeta a sincronizar
mkdir /home/nolberto/syncfolder
cd /home/nolberto/syncfolder

# Creo 100 archivos de prueba
for i in {1..100}; do touch file$i.txt; done

# Verifico que los 100 archivos fueron creados
ls | wc -l

# Genero clave SSH para conexión sin contraseña
ssh-keygen

# Copio la clave al servidor secundario (192.168.0.198)
ssh-copy-id nolberto@192.168.0.198

# Realizo una sincronización manual de prueba
rsync -avz /home/nolberto/syncfolder/ nolberto@192.168.0.198:/home/nolberto/syncfolder/

# Creo un script para automatizar la sincronización
nano /home/nolberto/sync.sh

#!/bin/bash

¡Entendido, Nolberto! No te preocupes, yo tomo el relevo desde aquí. El error que viste es común en Oracle Linux, ya que los paquetes de Alta Disponibilidad (HA) están en un repositorio que viene desactivado por defecto.

Aquí tienes la guía completa y corregida, integrando la solución de los repositorios para que solo tengas que copiar y pegar.

Práctica 2 — Cluster de Alta Disponibilidad

DATOS DEL CLUSTER
Nodo 1: m20252263 (192.168.0.199)

Nodo 2: vm20252263 (192.168.0.198)

IP Flotante: 192.168.0.200

Password hacluster: Norlin123n

PASO 1 — EN AMBOS NODOS: Habilitar Repositorios y Hostname
Copia y pega esto en ambos nodos para activar los paquetes de High Availability:

Bash
# Configurar hostname según el nodo (Asegúrate de estar en el correcto)
# En Nodo 1 usa: sudo hostnamectl set-hostname m20252263
# En Nodo 2 usa: sudo hostnamectl set-hostname vm20252263
exec bash

# Habilitar repositorios de Oracle Linux para HA
sudo dnf install -y oracle-epel-release-el9
sudo dnf config-manager --enable ol9_addons ol9_appstream ol9_baseos_latest
PASO 2 — EN AMBOS NODOS: Instalación de Paquetes
Ahora que los repos están activos, instala las herramientas del cluster:

Bash
sudo dnf install -y pacemaker corosync pcs fence-agents-all
PASO 3 — EN AMBOS NODOS: Configuración de Red y Seguridad
Preparamos el entorno para que los nodos se comuniquen sin bloqueos:

Bash
# Agregar hosts
sudo bash -c 'cat >> /etc/hosts << EOF
192.168.0.199  m20252263
192.168.0.198  vm20252263
EOF'

# Deshabilitar Firewall y SELinux (necesario para laboratorios de Redes)
sudo systemctl disable --now firewalld
sudo setenforce 0
sudo sed -i 's/^SELINUX=.*/SELINUX=disabled/' /etc/selinux/config

# Activar el servicio de configuración del cluster
sudo systemctl enable --now pcsd
PASO 4 — EN AMBOS NODOS: Contraseña de Cluster
Establece la contraseña para el usuario del sistema que gestiona el cluster:

Bash
echo "Norlin123n" | sudo passwd --stdin hacluster
PASO 5 — SOLO EN EL NODO 1 (m20252263): Crear el Cluster
Ejecuta esto solo desde tu primer nodo para unir a ambos:

Bash
# 1. Autenticar nodos
sudo pcs host auth m20252263 vm20252263 -u hacluster -p Norlin123n

# 2. Configurar y arrancar el cluster
sudo pcs cluster setup practica2 m20252263 vm20252263 --force
sudo pcs cluster start --all
sudo pcs cluster enable --all

# 3. Configuraciones globales (Importante: STONITH se desactiva si no tienes hardware de esgrima)
sudo pcs property set stonith-enabled=false
PASO 6 — SOLO EN EL NODO 1: Crear la IP Flotante
Este es el recurso que "brincará" de un nodo a otro si uno falla:

Bash
sudo pcs resource create virtual_ip \
  ocf:heartbeat:IPaddr2 \
  ip=192.168.0.200 \
  cidr_netmask=24 \
  nic=eth0 \
  op monitor interval=10s
PASO 7 — VERIFICACIÓN Y PRUEBAS
Para confirmar que todo funciona, usa:

Bash
sudo pcs status
Para probar el Failover:

En tu Windows abre una terminal y deja corriendo un ping -t 192.168.0.200.

En el Nodo 1, ejecuta: sudo pcs node standby m20252263.

Verás que el ping se pierde un segundo y vuelve, porque el Nodo 2 tomó la IP.

Para regresar a la normalidad: sudo pcs node unstandby m20252263.


🚀 PRÁCTICA 3: CLUSTER HTTP CON KEEPALIVED
DATOS DEL ESCENARIO
Nodo 1 (Master): m20252263 (192.168.0.199)

Nodo 2 (Backup): vm20252263 (192.168.0.198)

Virtual IP (VIP): 192.168.0.210 (Usaremos la .210 para no chocar con la práctica anterior)

Interfaz: eth0

PASO 1 — EN AMBOS NODOS: Instalar Apache y Keepalived
Bash
# Instalar los servicios
sudo dnf install -y httpd keepalived

# Habilitar e iniciar Apache
sudo systemctl enable --now httpd
PASO 2 — DIFERENCIAR LOS SERVIDORES (Página HTML)
Ejecuta esto en el Nodo 1 (m20252263):

Bash
echo "<h1>Bienvenido a SERVER 1 - m20252263</h1>" | sudo tee /var/www/html/index.html
Ejecuta esto en el Nodo 2 (vm20252263):

Bash
echo "<h1>Bienvenido a SERVER 2 - vm20252263</h1>" | sudo tee /var/www/html/index.html
PASO 3 — CONFIGURAR KEEPALIVED
En el NODO 1 (Master):
Borra todo el contenido de /etc/keepalived/keepalived.conf y pega esto:

Bash
sudo bash -c 'cat > /etc/keepalived/keepalived.conf << EOF
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass Norlin123n
    }
    virtual_ipaddress {
        192.168.0.210
    }
}
EOF'
En el NODO 2 (Backup):
Borra todo el contenido de /etc/keepalived/keepalived.conf y pega esto:

Bash
sudo bash -c 'cat > /etc/keepalived/keepalived.conf << EOF
vrrp_instance VI_1 {
    state BACKUP
    interface eth0
    virtual_router_id 51
    priority 90
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass Norlin123n
    }
    virtual_ipaddress {
        192.168.0.210
    }
}
EOF'
PASO 4 — INICIAR EL SERVICIO
Ejecuta esto en ambos nodos:

Bash
sudo systemctl enable --now keepalived
PASO 5 — VALIDACIÓN (La prueba del examen)
Verificar la IP: Ejecuta ip add show eth0 en el Nodo 1. Deberías ver la IP 192.168.0.210 asignada.

Navegador: Desde tu PC real (Windows), abre el navegador y entra a http://192.168.0.210. Debería decir: "SERVER 1".

Simular Falla: Apaga el Nodo 1 o detén el servicio:

Bash
sudo systemctl stop keepalived
Refrescar Navegador: Entra otra vez a http://192.168.0.210. Ahora debería decir: "SERVER 2".
